# Event-Page Persistence Plan

## Purpose

Store reviewed event-page links only after candidate quality is high enough.

This plan is design-only. It does not create tables, run migrations, or write event-page links to the database.

## Why Not Store All Candidates

Heuristic candidates can introduce noisy links. A location phrase can produce a valid page title that is still weak as an event-page link. Persistence should begin with reviewed candidates so later measurements remain auditable.

## Candidate Classes

- `registry_reviewed`: generated from `reference_candidate_registry.json` with `review_status=reviewed`.
- `registry_provisional`: generated from the registry but still awaiting additional review.
- `heuristic_fallback`: generated by location parsing or broad context rules outside the reviewed registry.

## Save Policy

Store by default:

- `registry_reviewed`
- `exists=true`
- confidence `high` or `medium`

Do not store by default:

- `heuristic_fallback`
- confidence `low`
- `exists=false`
- `ambiguous_location`

Store later with review:

- `registry_provisional`
- historical/context pages
- low-confidence registry links that have been explicitly approved

## Proposed Tables, Not Yet Executed

These tables are a proposal only. No SQL migration is included in this phase.

Suggested uniqueness:

- `measured_at`

## Write Flow Proposal

1. Generate candidate pages for selected normalized events.
2. Check page existence.
3. Classify candidates using the save policy.
4. Persist only `store_by_default` links.
5. Persist pageview-window measurements only for stored links.
6. Keep provisional and heuristic candidates in dry-run reports until reviewed.

## Open Questions

- Should persistence start with only reviewed links, or include a separate review queue?
- Should the database store daily pageviews or only summary windows?
- How should non-Wikipedia reference sources be represented later?
- Should `registry_provisional` candidates become persisted only after a manual review step?
- Should low-confidence local pages ever be stored automatically?

## Current Recommendation

Begin database persistence only for `registry_reviewed`, `exists=true`, high/medium-confidence links. Keep provisional, low-confidence, and heuristic fallback links out of the initial persistence set.
