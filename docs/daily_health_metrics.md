# Daily Health Metrics

## Purpose

Define the internal metrics used to verify that World Pulse daily build remains healthy and observable. These metrics are for internal review only and are not exposed in the public UI.

## Daily Metrics

### 1. GitHub Actions Duration

**What it means**

The total wall-clock time from workflow start to the final commit-and-push step. Duration changes can reveal upstream API slowdowns, query regressions, or dependency issues.

**Normal range**

A rough baseline should be established after the first two weeks of stable runs. Any run that takes more than double the established baseline is worth inspecting.

**When to investigate**

- Duration grows steadily over multiple days.
- Duration exceeds three times the established baseline.
- The workflow terminates with a timeout rather than success.

---

### 2. Daily Build Success

**Expected**

The workflow should succeed on every scheduled run (09:30 JST / 00:30 UTC).

**Failure handling**

- If one run fails, check the GitHub Actions log immediately.
- If two or more consecutive runs fail, treat this as a critical issue.
- Do not manually post to X while the build is in a failure state.
- See `failure_modes.md` for response procedures.

---

### 3. Data Date Consistency

**Expected data date**

UTC yesterday at the time the workflow runs.

**Actual data date**

The date embedded in `public/display/latest.json` and the generated share text.

**Mismatch handling**

- A one-day mismatch may indicate a timezone parsing issue or target_date logic error.
- Log the mismatch and identify whether it is systematic or a one-time occurrence.
- Do not post a share image whose data date does not match the expected UTC yesterday.

---

### 4. Source Lineage Completeness

**Expected**

The count of raw observations with no matching `source_lineage` record should be zero for each ingestion run.

**Check description**

Run the orphan count query from `automation_operation.md` after each ingestion. A non-zero count means some observations were not logged through the ingestion pipeline.

**Handling**

- Investigate the source that produced the orphan rows.
- Do not proceed with persistence steps if lineage is incomplete.

---

### 5. Normalized Events Count

**Daily count**

Track the daily count of normalized events generated by the build. Counts will vary by category and source availability.

**Unusual drop handling**

- A sharp drop may indicate a source outage or ingestion logic issue.
- Compare against the prior 7-day range.
- If the count drops to zero for a source that is normally active, treat this as a build health issue.

**Unusual spike handling**

- A large spike may indicate a deduplication issue or ingestion loop.
- Cross-check against raw observation counts and source lineage timestamps.

---

### 6. NOAA Kp / X-ray Coverage

**Tracked fields**

For each NOAA source, check:

- Earliest `observed_at` date in stored snapshots.
- Latest `observed_at` date in stored snapshots.
- Count of distinct observed days (usable days).

**Accumulation baseline**

At Phase 3 dry-run start:

- Kp usable days: approximately 2.42 days.
- X-ray usable days: approximately 9.17 days.

**Why accumulation matters**

NOAA provides only a current-window snapshot per fetch. Usable day counts grow only through repeated daily ingestion over time. Space Weather dry-run re-evaluations at 30, 60, and 90 days depend entirely on this accumulated history. If daily Kp or X-ray ingestion stalls, accumulated coverage stalls with it.

**When to check**

- Verify at least once per week that usable day count is increasing.
- If usable days do not grow over multiple consecutive days, check the NOAA ingestion step in the build log.

---

### 7. Share Image Integrity

**File exists check**

Confirm the following files are present after each build:

- `public/share/world-pulse-latest.jpg`
- `public/share/world-pulse-latest.png`
- `public/share/world-pulse-YYYY-MM-DD.jpg`
- `public/share/world-pulse-YYYY-MM-DD.png`

**File size check**

Confirm each file has size greater than zero. A zero-byte image indicates a generation failure.

**Rendering check**

Basic visual inspection of the share image remains manual. Automated pixel-level checks are not implemented in this phase.

---

### 8. X Dry-Run Readiness

Check `scripts/post_to_x.py` in dry-run mode after each build:

- `would_post`: confirms that the script sees valid text and image files and would create a post.
- `duplicate_post`: confirms that the data date has not already been posted (duplicate prevention is working).
- `blocked_wording`: confirms the generated text does not include any phrase on the blocked list.

A clean dry-run result means live posting could proceed if credentials and approval were in place. A failure in dry-run mode should be investigated before considering live posting.

---

## Weekly Review Metrics

Review the following items once per week:

- Source coverage: which sources produced observations in the past 7 days.
- API failure count: if any source returned non-2xx responses, count and identify them.
- Supabase storage trend: total rows and storage consumption direction.
- Strongest internal examples: review whether any new Kp >= 5 or X-ray M-class events were captured in NOAA snapshots.
- X dry-run consistency: confirm that text, image, and data date are consistent across multiple dry-run runs.

Weekly review notes should be recorded in an internal log or scratch document, not committed to the repository unless they represent a confirmed finding.

---

## Future Automation

A script `scripts/record_daily_health.py` may be added in a future phase to automate collection of these metrics into a structured log. This script is not implemented in the current phase. Manual review based on the GitHub Actions log and database queries is sufficient for now.
